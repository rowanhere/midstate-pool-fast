System Overview

Midstate is a peer-to-peer electronic cash system engineered to reject data-center centralization. It abandons elliptic curve mathematics in favor of post-quantum hash functions, enforces sequential CPU mining, and natively obscures transaction amounts and origins.

It is designed to operate and validate the global state entirely on extreme edge hardware, allowing a full node to sync and run on a $15 Raspberry Pi Zero 2 W.

Deep Dive: Hardware & Protocol Limits

The protocol strictly bounds resource consumption to ensure viability on 512MB RAM devices.

  • Max Batch Commits: 2,000 per block.
  • Max Batch Reveals: 500 per block (limits signature verification load).
  • Max Batch Inputs/Outputs: 1,024 inputs and 10,000 outputs globally per block.
  • Max Mempool Size: 100MB hard cap to prevent OOM attacks from malicious peers flooding large STARK payloads.

Post-Quantum Cryptography

Midstate uses exactly zero elliptic curve mathematics. It relies strictly on the BLAKE3 hash function, rendering it immune to Shor's algorithm running on future quantum computers.

WOTS+ (Single-Use Keys)

Receiving keys and change outputs use Winternitz One-Time Signatures. Because hash-based signatures reveal parts of the private key during use, WOTS keys must never sign more than once.

Deep Dive: WOTS+ Parameters

Midstate implements WOTS+ with parameter w=16 (16 bits per digit).

  • The 256-bit message is divided into 16 message chains.
  • The checksum calculates Σ (65535 - d_i). The maximum checksum value is 1,048,560 (20 bits), requiring exactly 2 checksum chains.
  • Total chains: 18. Maximum chain depth: 65,535.
  • Signature size is highly compact: 18 * 32 = 576 bytes.

MSS (Reusable Addresses)

If a user requires a static, reusable address, Midstate wraps thousands of WOTS keys into a Merkle Signature Scheme (MSS) binary tree.

Deep Dive: MSS Implementation

The Master Public Key is the root of the Merkle tree. The maximum supported tree height is 20 (2^20 = 1,048,576 signatures). A standard height-10 tree generates in ~1-2 seconds on modern CPUs.

To prevent key-reuse vulnerabilities across reorgs or malicious coordinator attacks, the consensus verifier enforces the leaf_index against the cryptographic authentication path, explicitly marking the exact WOTS public key leaf as spent in the spent_addresses database table.

Deep Dive: HD Key Derivation

Wallets use BIP39 24-word mnemonics, which generate a 64-byte seed that is hashed via BLAKE3 into a 32-byte master seed. To maintain forward secrecy without BIP32 ECC scalar tweaking, deterministic keys are derived via strict domain separation:

WOTS Seed_i = BLAKE3("midstate/wots/v1" || MasterSeed || i)
MSS Seed_i  = BLAKE3("midstate/mss/v1"  || MasterSeed || i)

Mining & Consensus

Traditional Proof-of-Work allows ASICs to calculate millions of hashes in parallel, centralizing network control.

Midstate enforces a Verifiable Delay Function (VDF) style Sequential Hash. To mine a block, a node must calculate 1,000,000 BLAKE3 hashes in a strict sequence: x_i = BLAKE3(x_{i-1}). Because calculating the final hash requires the sequential output of every previous step, highly parallelized ASIC farms lose their advantage against standard consumer CPUs.

Deep Dive: SIMD Nonce Searching

While the 1,000,000 hash chain is strictly sequential, finding a valid starting nonce is parallelized locally. The node automatically detects CPU features on startup and utilizes SIMD registers (AVX2: 8-way, NEON: 4-way).

Because different nonces reach the target at different iteration depths, the SIMD implementation uses a Masking Trick. It loops up to the maximum iterations required by any lane in the batch, safely extracting a lane's state exactly when its target is reached, while forcing it to "ghost hash" harmlessly for the remainder of the loop to keep the vector registers synchronized.

ASERT Difficulty & Finality

Difficulty adjusts continuously on every 60-second block using the ASERT algorithm.

Deep Dive: ASERT Fixed-Point Math

ASERT adjusts difficulty based on absolute time elapsed since genesis, eliminating sliding-window time-warp exploits. It uses a 4-hour half-life (240 blocks).

Floating-point math is strictly banned in consensus due to cross-architecture variations. Midstate uses a 16.16 fixed-point Taylor polynomial approximation to calculate the exponential decay exactly the same way on every processor:

// 16.16 fixed-point Taylor polynomial for 2^x
let mut factor = 65536i64;
factor += (frac * 45426) >> 16;
factor += (frac * frac * 15746) >> 32;
factor += (frac * frac * frac * 3643) >> 48;
Deep Dive: Bayesian Finality Estimator

Midstate eliminates arbitrary confirmation counts (e.g., "wait 6 blocks"). Nodes locally observe network honesty (linear blocks vs orphaned blocks) and use a Beta-Binomial statistical model.

The estimator continuously calculates a dynamic safe_depth—the exact number of blocks required to guarantee a 1e-6 (1-in-1,000,000) risk of a successful chain reorganization by an attacker. It caps at a maximum of 10,000 blocks for safety under extreme adversarial conditions.

MEV Resistance & Privacy

Commit-Reveal Lifecycle

Standard mempools broadcast transaction details in cleartext, allowing miners to front-run trades (Miner Extractable Value). Midstate enforces a two-phase protocol.

1. Commit Hash
→
Mine Commit
→
Reveal Tx Data
→
Execute Spend
Deep Dive: Dynamic Commit PoW

To prevent attackers from flooding the mempool with zero-fee Commits, every Commit requires a Proof-of-Work spam nonce. This threshold scales dynamically based on local mempool congestion:

  • < 500 pending commits: 24 leading zero bits (~16M hashes).
  • 500 - 749 pending commits: 26 leading zero bits.
  • 750 - 899 pending commits: 28 leading zero bits.
  • >= 900 pending commits: 30 leading zero bits.

Commits expire from the global state after a strict Time-to-Live (TTL) of 1,000 blocks to prevent state bloat.

Hidden Balances (ZK-STARKs)

Midstate supports Confidential Transfers using ZK-STARKs. The OP_VERIFY_STARK opcode mathematically proves that v ∈ [0, 2^64) and inputs equal outputs without revealing the actual values on the public ledger.

Deep Dive: STARK Memory Bypass

STARK proofs (using Winterfell over the Goldilocks field) are ~128KB in size. Pushing this onto a standard SmallVec script stack would cause massive heap allocation spikes, crashing 512MB edge nodes.

During execution, the script VM intercepts the STARK payload directly from the transaction slice. The proof bytes bypass the active stack memory completely during OP_VERIFY_STARK execution, processing the entire validation natively in Rust before safely returning execution to the VM.

Native CoinJoin & Dandelion++

Because Midstate enforces strict power-of-2 denominations for standard UTXOs, coins are inherently indistinguishable. The P2P layer includes a built-in CoinJoin coordinator to mix transactions seamlessly.

Deep Dive: Mix Anti-Sybil & Dandelion++

CoinJoin Anti-Sybil: To prevent attackers from stalling mix sessions with dummy registrations, joining a mix requires a localized PoW challenge: hash(mix_id || coin_id || peer_id || nonce) >= 24 zero bits. Binding the peer_id prevents attackers from pre-computing nonces across identity rotations.

Dandelion++ Routing: Commits route through a "stem" phase before public broadcast. They are forwarded linearly to a single peer. At each hop, there is a 10% probability of "fluffing" (broadcasting to all peers). If a transaction sits in the stem pool for 30 seconds without being fluffed, the node broadcasts it locally as a failsafe.

State Management

Midstate discards the traditional unbounded UTXO database model. Unspent outputs and pending commitments are accumulated into a Sparse Merkle Tree (SMT), distilling the global state down to a single 32-byte cryptographic root.

Deep Dive: SMT Architecture & Caching

The UTXO accumulator is a 256-level Sparse Merkle Tree backed by an immutable im::OrdSet. To achieve O(log N) performance without overwhelming memory:

  • Top Cache: The top 16 levels are cached in memory via HashMap, strictly bounded to a mathematical maximum of 65,536 entries.
  • Dynamic Folding: The bottom 240 levels are folded dynamically on the fly when proofs are requested, using a highly optimized binary search (partition_point) over lexicographically sorted coin ID buckets.

Fast-Forward Syncing

Historical block batches are aggressively pruned. Validating nodes only retain lightweight headers and state snapshots.

Deep Dive: Snapshot Auth & Compact Filters

Snapshots: During a fast-forward sync, a node downloads a recent State Snapshot. To ensure authenticity, it cryptographically verifies that the 32-byte midstate hash inside the snapshot perfectly aligns with the historical header chain. The node will only trust a snapshot if there are PRUNE_DEPTH (1,000) sequential headers built on top of it.

Compact Filters: Light clients scan the chain locally using Golomb-Rice coded compact filters built natively into the block structure. Using parameter P=20, they achieve a strict False Positive Rate of 1 in 1,000,000, ensuring high privacy and low bandwidth.

Virtual Machine

Every UTXO in Midstate is locked by a compiled bytecode script. The VM executes witness inputs pushed onto the stack, then evaluates the script. Execution succeeds if and only if the final top-of-stack item is exactly [1] (The Clean Stack Rule).

Deep Dive: Consensus Limits & Opcodes
  • Execution Limits: Turing-incomplete. No loops, no backward jumps. Max script size: 1,024 bytes. Max stack depth: 64 items. Max item size: 131,072 bytes (accommodates STARK proofs).
  • SigOps: Max 3 cryptographic signature verifications per script. Max 1 STARK verification per script.
  • Opcodes: Includes standard control flow (IF/ELSE), hashing, math, plus advanced covenants (CHECKTIMEVERIFY, SUM_TO_ADDR, VERIFY_STARK).

Script Architecture: Midstate uses a Pay-to-Script-Hash (P2SH) model where the address is the hash of the script bytecode. By default, wallets construct a Pay-to-Public-Key (P2PK) script where the raw public key is embedded directly into the script, rather than a P2PKH model.

Worked Example: Standard P2PK 
Script:  PUSH_DATA(<owner_pk>) CHECKSIGVERIFY PUSH_INT(1)
Witness: [ <signature> ]

Execution Trace:
1. Stack: [ <signature> ]
2. PUSH_DATA(<owner_pk>)
   Stack: [ <signature>, <owner_pk> ]
3. CHECKSIGVERIFY 
   (Pops pk, pops sig. Verifies signature against commitment. Aborts if invalid.)
   Stack: []
4. PUSH_INT(1)
   Stack: [ 1 ]

Result: VALID (Clean Stack Rule satisfied).
Worked Example: Hash Time-Locked Contract

HTLCs enable trustless atomic swaps. Coins can be claimed using a secret preimage, or refunded after a specific block height.

Script: 
IF 
    HASH PUSH_DATA(<secret_hash>) EQUALVERIFY 
    PUSH_DATA(<receiver_pk>) CHECKSIGVERIFY 
ELSE 
    DROP 
    PUSH_INT(<timeout_height>) CHECKTIMEVERIFY 
    PUSH_DATA(<refund_pk>) CHECKSIGVERIFY 
ENDIF 
PUSH_INT(1)

Claim Witness:  [ <receiver_sig>, <secret_preimage>, 1 ]
Refund Witness: [ <refund_sig>, <dummy_data>, 0 ]
Worked Example: STARK-Enforced Covenants

MidstateScript can enforce constraints on where outputs are sent using SUM_TO_ADDR, combined with VERIFY_STARK for zero-knowledge proofs.

Script:
VERIFY_STARK 
PUSH_DATA(<required_addr>) 
SUM_TO_ADDR PUSH_INT(100) GREATER_OR_EQUAL VERIFY 
PUSH_INT(1)

Witness: [ <program_id>, <public_inputs>, <128KB_proof_payload> ]

Execution Trace:
1. Stack: [ <program_id>, <public_inputs> ]
2. VERIFY_STARK
   (Pops inputs, intercepts 128KB payload directly from tx memory. Aborts if invalid.)
   Stack: []
3. PUSH_DATA(<required_addr>)
   Stack: [ <required_addr> ]
4. SUM_TO_ADDR
   (Pops address, scans all tx outputs, sums the value sent to that address)
   Stack: [ <actual_sum_sent> ]
5. PUSH_INT(100)
   Stack: [ <actual_sum_sent>, 100 ]
6. GREATER_OR_EQUAL VERIFY
   (Validates that the tx sent AT LEAST 100 units to required address)
   Stack: []
7. PUSH_INT(1)
   Stack: [ 1 ]

Deep Dive
Legacy networks use secp256k1, which is vulnerable to Shor's algorithm on quantum computers. Midstate uses Winternitz One-Time Signatures (w=16, 576 bytes) and Merkle Signature Schemes (max height 20) which rely solely on the quantum-resistant BLAKE3 hash function. Mining Hardware Parallel ASICs / GPUs Sequential Hashing (CPUs)
Deep Dive
Legacy PoW allows parallel nonce searching, giving ASICs a massive advantage. Midstate enforces 1,000,000 strictly sequential BLAKE3 hashes per extension. SIMD lanes (AVX2/NEON) are only used to parallelize the search for valid starting nonces, while the proof calculation itself remains strictly bound to single-thread CPU performance.
Mempool / Ordering Cleartext Bidding (MEV) Commit-Reveal
Deep Dive
Legacy mempools expose transaction contents, allowing miners to reorder trades for profit (MEV). Midstate's Phase 1 (Commit) broadcasts a BLAKE3 hash binding inputs/outputs with a required dynamic PoW nonce. Phase 2 (Reveal) opens the commitment. Miners cannot reorder or censor transactions based on value or destination because they commit to processing them before seeing the contents.

Deep Dive
Legacy ledgers show exact amounts to the public. Midstate supports ZK-STARKs over the Goldilocks field to prove value conservation (inputs = outputs) and range constraints (v ∈ [0, 2^64)) without revealing amounts. Standard UTXOs use strict power-of-2 denominations to facilitate native P2P CoinJoin mixing with anti-Sybil PoW gating.

Deep Dive
Legacy nodes store every unspent output in an ever-growing database. Midstate accumulates UTXOs into a 256-level SMT (Top 16 levels cached in memory), distilling global state into a 32-byte root. This allows edge nodes (512MB RAM) to operate with O(log N) complexity and fast-forward sync from snapshots via cryptographic midstate linkage.
